Manual operacional do sistema

Este manual organiza o uso da Cyber-Nexis em camadas: acesso, navegação, chat, contato, manutenção dos arquivos e segurança básica. A regra central é simples: menos improviso, mais controle.

Camada 01

Guia rápido

Use este projeto como um site estático com autenticação Firebase, chat em tempo real e páginas de apoio. A estrutura foi padronizada para evitar erros comuns de GitHub Pages, principalmente 404 causado por nomes diferentes como Index.html, index.html, Styles.css e styles.css.

1. Acesso

O usuário entra por login.html. Depois do login, as páginas protegidas podem chamar protectPage().

2. Conteúdo

O portal principal fica em index.html. As páginas secundárias usam a mesma navegação e o mesmo CSS.

3. Comunicação

O chat fica em chat.html e usa chat.js para enviar mensagens com Firebase/Firestore.

Arquitetura recomendada

Estrutura dos arquivos

Mantenha os nomes em minúsculo, sem espaços e sem acentos. O GitHub Pages diferencia letras maiúsculas de minúsculas. Um link para Regras.html não encontra automaticamente regras.html. 

CYBER_NEXIS/
├── index.html
├── login.html
├── login.js
├── chat.html
├── chat.js
├── contato.html
├── contato.js
├── manual.html
├── regras.html
├── recrutamento.html
├── quiz.html
├── cyber-nexis-comando.html
├── firebase-config.js
├── site.js
├── particles-config.js
├── acesso-secreto.js
├── styles.css
└── Imagem/

Regra de ouro

Se o arquivo se chama styles.css, todos os HTMLs devem chamar exatamente styles.css.

Evite duplicações

Não mantenha duas versões do mesmo arquivo, como Index.html e index.html. Isso cria links falsos.

Camada Firebase

Configuração do Firebase

O Firebase deve ser centralizado em firebase-config.js. Login, chat e demais scripts devem importar a configuração desse arquivo, em vez de repetir chaves e versões diferentes em cada página.

Ative autenticação

No Firebase Console, ative Email/Password para login tradicional e Anonymous se quiser permitir chat anônimo.

Ative Firestore

O chat usa Firestore para salvar mensagens. Configure regras adequadas antes de publicar para outras pessoas.

Não duplique projetos

Evite usar um projeto Firebase no login e outro no chat, a menos que isso seja uma decisão consciente.

Chaves públicas

A chave do Firebase no front-end não é senha secreta, mas as regras do Firestore precisam estar bem configuradas.

Comunicação restrita

Uso do chat

O chat deve ter um único HTML e um único JS compatível. O erro antigo acontecia porque o HTML possuía IDs diferentes dos IDs esperados pelo JavaScript. Agora a estrutura deve permanecer sincronizada.

Sala pública

Use a sala pública para mensagens gerais e testes iniciais do sistema.

Sala privada

Use uma chave de sala para criar um ambiente separado. Não use senhas reais ou pessoais.

Anexos

Limite imagens antes de enviar. Arquivos muito grandes deixam o Firestore pesado e caro.

Segurança básica

Boas práticas

O site pode ter estética de sistema restrito, mas segurança real vem de regras, autenticação e código limpo. Efeito visual não substitui proteção técnica.

Evite innerHTML com dados do usuário

Mensagens, nomes e textos enviados por visitantes devem ser renderizados com textContent.

Não coloque senhas no JavaScript

Qualquer senha escrita em arquivo JS pode ser vista por quem abrir as ferramentas do navegador.

Proteja rotas importantes

Páginas internas podem usar protectPage("login.html") para bloquear visitantes sem login.

Revise regras do Firestore

Não publique com regras abertas permanentemente. Elas são a muralha real do banco de dados.

Canal externo

Página de contato

O formulário de contato usa contato.js. Para envio real com EmailJS, preencha corretamente os três valores: Public Key, Service ID e Template ID. Não use o mesmo valor para todos. 

emailjs.init("SUA_PUBLIC_KEY");
emailjs.send("SEU_SERVICE_ID", "SEU_TEMPLATE_ID", templateParams);

Antes de publicar, teste o formulário com nome, e-mail e mensagem válidos. O honeypot contra spam deve continuar vazio para usuários reais.

Publicação

Como subir no GitHub Pages

1. Suba na raiz

Coloque index.html diretamente na raiz do repositório, não dentro de uma pasta escondida.

2. Preserve a pasta de imagens

A pasta deve continuar como Imagem/, exatamente igual aos caminhos usados no HTML.

3. Ative Pages

Em Settings → Pages, publique pela branch main e pasta /root.

4. Teste links

Abra cada página: início, chat, manual, regras, recrutamento, quiz e contato. Link morto é ruído operacional.

Checklist final

Antes de publicar

Arquivos

Todos os nomes estão em minúsculo, exceto a pasta Imagem/ se ela já existir assim no repositório.

CSS

Todas as páginas chamam styles.css.

Firebase

Login e chat usam a mesma configuração central em firebase-config.js.

Links

Nenhum link aponta para Index (1).html, Regras.html ou arquivos que não existem.
